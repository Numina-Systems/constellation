# Web Tools Design

## Summary

This design adds two new tools to the agent — `web_search` and `web_fetch` — giving it the ability to query the web and retrieve page content at runtime. Both tools live in a new `src/web/` module and are registered in the existing `ToolRegistry`, making them available to the agent loop as native tool calls and to sandboxed Deno code via the IPC bridge, with no special-casing required in the agent loop itself.

Search is built on a `SearchProvider` port interface with four concrete adapters (Brave, Tavily, SearXNG, DuckDuckGo) composed into a sequential fallback chain — the chain tries providers in configured order, silently skips any that lack credentials, and returns the first successful result. Fetching runs HTML through a three-stage pipeline: raw retrieval, content extraction via Mozilla Readability (strips ads, nav, and chrome), and markdown conversion via Turndown with GFM support. Results are paginated by character offset and cached in memory for five minutes to avoid redundant network round-trips. The entire design mirrors established port/adapter patterns already in use for the model and embedding subsystems.

## Definition of Done
- The agent has a `web_search` tool that queries the web and returns structured results (title, URL, snippet/content), using a sequential fallback chain of configured providers (Brave → Tavily → SearXNG → DuckDuckGo)
- The agent has a `web_fetch` tool that retrieves a URL and returns its content as clean markdown, with pagination for large pages and a 5-minute cache
- Both tools are registered in the `ToolRegistry` and available to the agent as native tool calls AND from inside the Deno sandbox via IPC bridge stubs
- Search providers are behind a `SearchProvider` port interface, making it trivial to add new providers
- Configuration lives in a new `[web]` section in `config.toml` with API keys overridable via environment variables
- Existing Deno `--allow-net` behaviour is unchanged

## Acceptance Criteria

### web-tools.AC1: Search providers return structured results
- **web-tools.AC1.1 Success:** Brave adapter parses API response into SearchResponse with title, url, snippet for each result
- **web-tools.AC1.2 Success:** Tavily adapter parses API response into SearchResponse with title, url, snippet, and score
- **web-tools.AC1.3 Success:** SearXNG adapter parses JSON response into SearchResponse
- **web-tools.AC1.4 Success:** DuckDuckGo adapter parses HTML response into SearchResponse
- **web-tools.AC1.5 Failure:** Provider returns error when API responds with non-2xx status
- **web-tools.AC1.6 Failure:** Provider returns error when response body is unparseable
- **web-tools.AC1.7 Failure:** Provider returns error when request times out

### web-tools.AC2: Fetch pipeline converts URLs to paginated markdown
- **web-tools.AC2.1 Success:** Fetcher retrieves HTML page and returns markdown with headings, links, lists, and tables preserved
- **web-tools.AC2.2 Success:** Readability extracts article content from noisy pages (strips nav, ads, sidebars)
- **web-tools.AC2.3 Success:** Large content is paginated with offset and has_more fields; continue_from retrieves next chunk
- **web-tools.AC2.4 Success:** Fetched content is cached; second request within 5 minutes returns cached result without HTTP request
- **web-tools.AC2.5 Success:** Cache entries expire after TTL; request after expiry fetches fresh content
- **web-tools.AC2.6 Failure:** When Readability fails to extract, fetcher falls back to raw Turndown conversion
- **web-tools.AC2.7 Failure:** Non-HTML content type (PDF, image) returns error with content type indicated
- **web-tools.AC2.8 Failure:** Content exceeding max_fetch_size is truncated with truncation noted in result
- **web-tools.AC2.9 Failure:** Fetch timeout returns error

### web-tools.AC3: Fallback chain tries providers sequentially
- **web-tools.AC3.1 Success:** Chain calls providers in configured order; first success is returned
- **web-tools.AC3.2 Success:** Unconfigured providers (missing API key/endpoint) are silently skipped
- **web-tools.AC3.3 Success:** Response includes provider name indicating which provider answered
- **web-tools.AC3.4 Failure:** When all providers fail, error lists each provider and its failure reason

### web-tools.AC4: Tools registered and available via IPC bridge
- **web-tools.AC4.1 Success:** web_search and web_fetch appear in registry.getDefinitions() when [web] config exists
- **web-tools.AC4.2 Success:** web_search and web_fetch are absent from registry when [web] config is omitted
- **web-tools.AC4.3 Success:** registry.generateStubs() produces typed IPC bridge functions for both tools

### web-tools.AC5: Configuration and environment overrides
- **web-tools.AC5.1 Success:** BRAVE_API_KEY and TAVILY_API_KEY environment variables override config.toml values
- **web-tools.AC5.2 Success:** Deno --allow-net and --deny-net behaviour is unchanged by web tool addition

## Glossary

- **Agent loop**: The core inference cycle that sends the conversation context to the LLM, receives a response, executes any requested tool calls, and appends results before the next turn.
- **Barrel export**: An `index.ts` file that re-exports the public API of a module, so consumers import from the module root rather than individual files.
- **Composition root**: `src/index.ts` — the single location where all dependencies are wired together. The only place that constructs concrete adapters and registers tools.
- **Deno**: A secure JavaScript/TypeScript runtime used as the sandboxed subprocess for `execute_code`. Unlike Bun (the host runtime), it runs with explicit permission flags (`--allow-net`, `--deny-net`, etc.).
- **Fallback chain**: The `SearchChain` — an ordered list of `SearchProvider` adapters tried sequentially; the first to succeed wins, the rest are skipped if unconfigured.
- **Functional Core / Imperative Shell**: Architectural pattern where pure data-transformation logic (Functional Core) is kept separate from code with side effects like I/O (Imperative Shell). Each source file declares which pattern it follows.
- **GFM (GitHub Flavoured Markdown)**: A Markdown dialect that adds tables, task lists, and fenced code blocks. The Turndown GFM plugin ensures these are preserved during HTML-to-markdown conversion.
- **Hexagonal architecture (port/adapter)**: A structural pattern where the domain defines abstract interfaces (ports) and concrete implementations (adapters) are swapped in at the boundary. Used throughout Constellation for models, embeddings, and now search providers.
- **IPC bridge**: The inter-process communication channel between the Bun host and the Deno sandbox. Tool calls made from inside sandboxed code are serialised over this channel and executed on the host.
- **linkedom**: A pure-JavaScript DOM implementation used to parse HTML outside a browser environment. Required by Mozilla Readability, which expects a DOM tree.
- **Mozilla Readability**: A library (originally from Firefox Reader View) that extracts the primary article content from an HTML page, stripping surrounding navigation, ads, and boilerplate.
- **Port interface**: A TypeScript `interface` (or type with a known shape) that defines what a dependency must do without specifying how. `SearchProvider` is a port; `BraveAdapter` is an adapter implementing it.
- **SearXNG**: A self-hosted, open-source meta-search engine that aggregates results from multiple sources and exposes a JSON API. Requires a user-operated instance.
- **SearchChain**: The concrete implementation of the fallback chain — iterates configured `SearchProvider` adapters in order, returning the first success or an aggregated error.
- **ToolRegistry**: The central registry that stores registered tools, validates parameters, dispatches calls to handlers, and generates Deno IPC stubs. Both `web_search` and `web_fetch` are registered here.
- **ToolResult**: The standard return type for all tool handlers — `{ success, output, error? }`. Handlers never throw; errors are captured in this struct.
- **Turndown**: A JavaScript library that converts HTML to Markdown. Used as the final stage of the fetch pipeline.
- **TTL (Time To Live)**: The duration a cache entry is considered valid. Here, 5 minutes for fetched page content.
- **Zod**: A TypeScript-first schema validation library used throughout the config subsystem. `WebConfigSchema` is a new Zod schema added to the existing `AppConfigSchema`.

## Architecture

Two native tools — `web_search` and `web_fetch` — registered in the `ToolRegistry` and backed by a new `src/web/` module following the project's hexagonal port/adapter pattern.

### Search

`web_search` delegates to a `SearchProvider` port interface. Multiple provider adapters (Brave, Tavily, SearXNG, DuckDuckGo) are composed into a sequential fallback chain. On invocation, the chain iterates configured providers in order, skipping any that lack required configuration (e.g. Brave without an API key). The first provider to return results wins. If all fail, the tool returns an error listing each provider and its failure reason.

```typescript
type SearchResult = {
  title: string;
  url: string;
  snippet: string;
  score?: number;
};

type SearchResponse = {
  results: ReadonlyArray<SearchResult>;
  provider: string;
};

interface SearchProvider {
  readonly name: string;
  search(query: string, limit: number): Promise<SearchResponse>;
}
```

### Fetch

`web_fetch` retrieves a URL and converts its content to markdown through a three-stage pipeline:

1. **Fetch** — HTTP GET via `fetch()` with configurable timeout and content-size limit
2. **Extract** — Mozilla Readability (via linkedom DOM parser) strips navigation, ads, and chrome to isolate article content. Falls back to raw HTML on extraction failure.
3. **Convert** — Turndown with GFM plugin converts extracted HTML to markdown preserving headings, links, lists, tables, and code blocks

Results are paginated by character offset for large pages. A 5-minute in-memory cache (keyed by URL) avoids redundant fetch/extract/convert cycles.

```typescript
type FetchResult = {
  url: string;
  title: string;
  content: string;
  total_length: number;
  offset: number;
  has_more: boolean;
};
```

### Data Flow

```
Agent loop
  ├─ web_search(query, limit?)
  │    └─ SearchChain → Provider1 → Provider2 → ... → SearchResponse
  └─ web_fetch(url, continue_from?)
       └─ Cache hit? → paginate cached markdown
          Cache miss? → fetch → Readability → Turndown → cache → paginate
```

Both tools dispatch through the standard `ToolRegistry`. No special-case handling in the agent loop (unlike `execute_code`). Both are also available from inside the Deno sandbox via IPC bridge stubs generated by `registry.generateStubs()`.

## Existing Patterns

This design follows established patterns from the codebase:

- **Port/adapter boundaries** — `SearchProvider` mirrors the structure of `ModelProvider` in `src/model/types.ts` and `EmbeddingProvider` in `src/embedding/types.ts`. Domain types in `types.ts`, adapter implementations in separate files.
- **Factory functions** — `createSearchChain(config)` and `createWebTools(config, chain)` follow the `createToolRegistry()`, `createMemoryTools(manager)` pattern. No classes.
- **Tool registration** — Tools are created and registered in `src/index.ts` during composition, same as memory tools. Conditional registration (only when `[web]` config section exists) follows the optional dependency pattern.
- **Config schema** — New `WebConfigSchema` added to `AppConfigSchema` with Zod validation and env var overrides, following the existing pattern in `src/config/config.ts`.
- **Error handling** — Tool handlers return `ToolResult` (never throw). Errors captured in `error` field with human-readable messages.

No divergence from existing patterns. The `src/web/` module structure is new but mirrors `src/model/` and `src/embedding/`.

## Additional Considerations

**Non-HTML content:** `web_fetch` returns an error for non-HTML content types (PDF, images, binary). The error message indicates the content type so the agent can decide whether to try an alternative approach (e.g. downloading via `execute_code`).

**DuckDuckGo fragility:** The DDG adapter scrapes HTML, which can break when DDG changes their markup. It's positioned last in the default fallback chain for this reason. If it breaks, the other three providers still work and DDG can be fixed or removed without impact.

**Cache eviction:** The in-memory cache uses lazy eviction (check TTL on access, don't proactively sweep). For a daemon that runs continuously, this means stale entries linger until re-requested. Acceptable for the expected access pattern; if memory becomes a concern, a periodic sweep can be added later.
