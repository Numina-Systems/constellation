# Skills System for Constellation

## Context

Constellation's agent currently has tools and memory but no concept of **reusable behavioural guides** — structured instructions that teach the agent *how* to approach specific situations. Inspired by open-strix's skills system (markdown files with YAML frontmatter, discovered dynamically, injected into context when relevant), we're building a skills system that:

- Uses **embedding-based retrieval** (not system-prompt enumeration) to find relevant skills per turn
- Supports **both human-authored and agent-authored** skills
- Is primarily **instructional markdown** but can optionally **register tool definitions**
- Fits cleanly into constellation's existing Functional Core / Imperative Shell architecture

## Module Structure

New module at `src/skill/` following the same port/adapter pattern as `src/memory/`:

```
src/skill/
  types.ts            # Functional Core — domain types
  parser.ts           # Functional Core — YAML frontmatter + body parsing
  parser.test.ts
  store.ts            # Functional Core — SkillStore port interface
  postgres-store.ts   # Imperative Shell — pgvector adapter
  loader.ts           # Imperative Shell — filesystem scan + embedding lifecycle
  loader.test.ts
  registry.ts         # Imperative Shell — SkillRegistry implementation
  registry.test.ts
  tools.ts            # Imperative Shell — skill_create/update/list/read agent tools
  tools.test.ts
  index.ts            # barrel exports
```

Built-in skills directory at project root:

```
skills/               # peer to src/, checked into repo
  <skill-name>/
    SKILL.md
    <companion>.md    # optional companion files
```

User/agent-authored skills in a configurable directory (default: `./user-skills/`).

## Skill File Format

```markdown
---
name: onboarding
description: >
  Guide for initial agent setup. Use when no core:persona block exists
  or it contains the default placeholder. Do not use for routine conversations.
version: "1.0"
tags:
  - setup
  - identity
companions:
  - establishing-identity.md
tools:
  - name: generate_persona_template
    description: Generate a persona template based on traits
    parameters:
      - name: traits
        type: string
        description: Comma-separated personality traits
        required: true
---

# Onboarding

Markdown body with instructions...
```

Frontmatter validated with Zod:
- `name`: required, kebab-case (`/^[a-z0-9-]+$/`)
- `description`: required, max 500 chars — must include "use when" and "do not use for" triggers
- `version`: optional string
- `tags`: optional string array
- `companions`: optional string array (relative paths to companion files)
- `tools`: optional array of tool definitions (reuses `ToolParameter` shape from `src/tool/types.ts`)

## Implementation Phases

### Phase 1: Types + Parser + Store Port

**New files:**
- `src/skill/types.ts` — `SkillMetadata`, `SkillDefinition`, `SkillSource`, `SkillSearchResult`, `SkillRegistry` interface
- `src/skill/parser.ts` — Pure `parseSkillFile(content) -> ParseResult` function. Splits on `---` frontmatter delimiters, validates with Zod
- `src/skill/parser.test.ts` — Parser unit tests (valid/invalid frontmatter, missing fields, malformed YAML)
- `src/skill/store.ts` — `SkillStore` port interface (`upsertEmbedding`, `deleteEmbedding`, `getByHash`, `searchByEmbedding`)
- `src/skill/index.ts` — Initial barrel

**Key types:**
```typescript
type SkillSource = 'builtin' | 'user';

type SkillDefinition = {
  readonly id: string;           // `skill:${source}:${name}`
  readonly metadata: SkillMetadata;
  readonly body: string;
  readonly companions: ReadonlyArray<{ name: string; content: string }>;
  readonly source: SkillSource;
  readonly filePath: string;
  readonly contentHash: string;  // SHA-256 of full SKILL.md for change detection
};

interface SkillRegistry {
  load(): Promise<void>;
  getAll(): Array<SkillDefinition>;
  getByName(name: string): SkillDefinition | undefined;
  search(query: string, limit?: number): Promise<Array<SkillSearchResult>>;
  getRelevant(context: string, limit?: number, threshold?: number): Promise<Array<SkillDefinition>>;
  createUserSkill(name: string, description: string, body: string, tags?: ReadonlyArray<string>): Promise<SkillDefinition>;
  updateUserSkill(name: string, description: string, body: string, tags?: ReadonlyArray<string>): Promise<SkillDefinition>;
}
```

**New dependency:** `yaml` npm package for YAML parsing (small, zero-dep).

### Phase 2: Database + Postgres Store

**New files:**
- `src/persistence/migrations/003_skill_embeddings.sql`
- `src/skill/postgres-store.ts`

**Migration:**
```sql
CREATE TABLE skill_embeddings (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    description TEXT NOT NULL,
    content_hash TEXT NOT NULL,
    embedding vector,
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_skill_embeddings_name ON skill_embeddings (name);
```

This table is a **search index only** — source of truth is always the SKILL.md files on disk. Can be dropped and rebuilt from filesystem at any time.

**Pattern to follow:** `src/memory/postgres-store.ts` — same `PersistenceProvider` injection, same `pgvector` usage with `toSql()` for embedding vectors, same cosine distance operator (`<=>`).

### Phase 3: Loader + Registry

**New files:**
- `src/skill/loader.ts` — `loadSkills(options) -> LoadResult`
- `src/skill/loader.test.ts`
- `src/skill/registry.ts` — `createSkillRegistry(options) -> SkillRegistry`
- `src/skill/registry.test.ts`

**Loader algorithm:**
1. Scan `builtinDir` for `*/SKILL.md` files, parse each → `SkillDefinition` with `source: 'builtin'`
2. Scan `userDir` for `*/SKILL.md` files, parse each → `SkillDefinition` with `source: 'user'`
3. Name conflicts: user skills override builtin (user explicitly wants to replace behaviour)
4. For each skill, compute `contentHash` (SHA-256). Check `store.getByHash(id)`. If hash matches, skip (already embedded). If not, embed and upsert
5. Orphan cleanup: delete embeddings for IDs no longer on disk
6. Load companion files referenced in `companions` metadata, relative to the SKILL.md directory

**What gets embedded:** `description` + tags + first ~500 chars of body. Enough signal for semantic matching without diluting with the full (potentially very long) guide body.

**Registry:**
- `load()` calls `loadSkills()`, stores results in `Map<string, SkillDefinition>`
- `getRelevant(context, limit, threshold)` — the per-turn entrypoint. Embeds the user message, calls `store.searchByEmbedding()`, resolves IDs to in-memory definitions, returns top-N above threshold
- `createUserSkill()` / `updateUserSkill()` — write SKILL.md to disk, parse, embed, add to in-memory map

### Phase 4: Agent Integration

**Modified files:**
- `src/agent/types.ts` — Add `skills?: SkillRegistry` to `AgentDependencies`. Add `max_skills_per_turn?: number` and `skill_threshold?: number` to `AgentConfig`
- `src/agent/context.ts` — `buildSystemPrompt(memory, skills?)` accepts optional skill definitions and appends a `## Active Skills` section after core memory blocks
- `src/agent/agent.ts` — Before the tool loop, query `deps.skills?.getRelevant(userMessage, config.max_skills_per_turn, config.skill_threshold)` and pass results to `buildSystemPrompt`

**Context budget:** Skills are injected into the system prompt string. `estimateTokens` and `shouldCompress` already operate on the full system prompt, so skill content is automatically accounted for in context budget calculations. `max_per_turn` (default: 3) and `similarity_threshold` (default: 0.3) keep injection bounded.

**Skill-defined tools:** When the registry loads, any skills with `tools` in their metadata get those tools registered in the `ToolRegistry`. The handler returns the skill's markdown body as instructions — the LLM reads the instructions and uses existing tools (memory, execute_code, etc.) to carry them out. This keeps skills purely declarative.

### Phase 5: Skill Authoring Tools + Composition Root

**New files:**
- `src/skill/tools.ts` — `createSkillTools(registry) -> Array<Tool>`
- `src/skill/tools.test.ts`

**Tools:**
| Tool | Description |
|------|-------------|
| `skill_list(source?)` | List all skills with names, descriptions, sources, tags |
| `skill_read(name)` | Read full skill content including companions |
| `skill_create(name, description, body, tags?)` | Create new user skill |
| `skill_update(name, description, body, tags?)` | Update existing user skill (user skills only) |

**Composition root changes (`src/index.ts`):**
```typescript
// After web tools registration, before agent creation:
if (config.skills) {
  const skillStore = createPostgresSkillStore(persistence);
  const skillRegistry = createSkillRegistry({
    store: skillStore,
    embedding,
    builtinDir: config.skills.builtin_dir,
    userDir: config.skills.user_dir,
  });
  await skillRegistry.load();

  // Register skill management tools
  for (const tool of createSkillTools(skillRegistry)) {
    registry.register(tool);
  }

  // Register skill-defined tools
  for (const skill of skillRegistry.getAll()) {
    for (const toolDef of skill.metadata.tools ?? []) {
      registry.register({
        definition: { name: toolDef.name, description: toolDef.description, parameters: toolDef.parameters },
        handler: async () => ({
          success: true,
          output: `[Skill: ${skill.metadata.name}]\n\n${skill.body}`,
        }),
      });
    }
  }
}

// Pass to agent:
const agent = createAgent({
  ...existingDeps,
  skills: skillRegistry ?? undefined,
});
```

## Boundaries: Skills vs Memory

| | Memory | Skills |
|---|---|---|
| Source of truth | Database | Filesystem (SKILL.md) |
| In context | Core: always, Working: always | Only when relevant (top-N per turn) |
| Embedding purpose | Recall via search | Per-turn relevance matching |
| Owned by | Agent (owner-scoped) | System (shared across conversations) |
| Content type | Freeform text | Structured markdown with frontmatter |

Skills teach the agent *how to do things*. Memory stores *what the agent knows*. A skill can reference memory tools, but they're separate concerns.
