# Hybrid Search + Multi-Domain Memory Retrieval Design

## Summary

Constellation's agent currently has access to its memories and conversation history, but no direct way to search across them with precision. This design adds a `search` tool that lets the agent query both its memory blocks and past conversations using keyword matching, semantic (vector) similarity, or a hybrid of both — and get back a single ranked list of results regardless of which data source they came from.

The implementation introduces a `SearchStore` port with a pluggable `SearchDomain` interface, following the same port/adapter boundaries used throughout the codebase. Each domain (memory, conversations) handles its own SQL query logic internally — running keyword and vector searches as CTEs within a single database query — and returns ranked candidates. The `SearchStore` then merges results across domains using Reciprocal Rank Fusion (RRF), a rank-combining algorithm that rewards results appearing in multiple retrieval signals. To support semantic search over conversations, message embeddings are generated at insert time and backfilled for existing messages. The design is built for extensibility: adding a new search domain (e.g., cross-agent messages) requires only implementing the `SearchDomain` interface and registering it — no changes to the core merge or tool logic.

## Definition of Done

1. **A new `search` tool** available to the agent with `mode` (semantic/keyword/hybrid), `domain` (memory/conversations/all), optional time range, optional role filter, and result limit
2. **Hybrid search infrastructure** — tsvector columns + GIN indexes on both `memory_blocks` and `messages` tables, with RRF fusion combining keyword and vector results
3. **Message embeddings** — migration adding embedding column to `messages`, embedding generation on insert, and a backfill mechanism for existing messages
4. **Multi-domain search** — fan-out across memory blocks and conversations with unified scoring, designed to be extensible to a future cross-agent message domain

## Glossary

- **pgvector**: A PostgreSQL extension that adds a `vector` column type and approximate nearest-neighbour operators (like `<=>` for cosine distance), enabling vector similarity search directly in SQL.
- **tsvector**: A PostgreSQL data type representing a preprocessed, lexeme-normalised document for full-text search. Stored alongside the original text and queried with `tsquery` expressions.
- **GIN index**: Generalised Inverted Index — a PostgreSQL index type suited for composite values like tsvector columns. Required for efficient full-text keyword search.
- **`GENERATED ALWAYS AS ... STORED`**: A PostgreSQL column definition that computes and persists a derived value automatically on every insert or update. Used here so `search_vector` is always in sync with `content` without application-layer involvement.
- **Cosine similarity**: A measure of similarity between two vectors based on the angle between them, ranging from 0 (orthogonal) to 1 (identical direction). Used to rank how semantically similar a query is to stored embeddings.
- **CTE (Common Table Expression)**: A named subquery block in SQL (introduced with `WITH`), used here to run keyword and vector searches as parallel, composable steps within a single query.
- **Reciprocal Rank Fusion (RRF)**: A rank aggregation algorithm that combines multiple ranked lists into one. Each result's score is `1 / (k + rank)` summed across lists, where `k=60` is a smoothing constant. Results appearing in multiple lists score higher than those in only one.
- **Fan-out**: Dispatching the same logical request to multiple independent backends concurrently (here via `Promise.all()`), then merging their responses.
- **Port/adapter boundary**: An architectural pattern where a `port` is a domain interface (e.g., `SearchStore`) and an `adapter` is the concrete implementation (e.g., `createSearchStore()` backed by PostgreSQL). The domain never depends on the implementation.
- **Embedding**: A fixed-length vector of floats produced by a language model that encodes the semantic meaning of a piece of text. Embeddings close together in vector space represent semantically similar content.
- **Three-tier memory**: Constellation's memory model: `core` (always in context), `working` (active session), and `archival` (long-term, searchable). Each tier has different retrieval and persistence behaviour.
- **HNSW index**: Hierarchical Navigable Small World — an approximate nearest-neighbour index structure for vector search. Faster than sequential scan at scale; deferred in this design until query performance warrants it.
- **Backfill**: A one-time data migration that retroactively applies a new processing step (here, embedding generation) to existing rows that predate the feature.
- **Composition root**: The single entry point (`src/index.ts`) where all dependencies are instantiated and wired together. The only place in the codebase where concrete implementations are chosen.
- **`ts_rank`**: A PostgreSQL function that scores a tsvector match against a tsquery based on term frequency and document coverage. Used as the relevance signal for keyword search results.

## Architecture

New `SearchStore` port with pluggable `SearchDomain` implementations. Each domain (memory, conversations) handles its own hybrid query logic internally. The `SearchStore` orchestrates fan-out to requested domains via `Promise.all()` and merges results using Reciprocal Rank Fusion (RRF, k=60).

### Components

**`src/search/` module:**
- `types.ts` — `SearchParams`, `SearchResult`, `DomainSearchResult`, `SearchDomain` interface
- `store.ts` — `SearchStore` port interface
- `postgres-store.ts` — `createSearchStore()` implementation with RRF merge logic
- `domains/memory.ts` — `MemorySearchDomain` (queries `memory_blocks`)
- `domains/conversations.ts` — `ConversationSearchDomain` (queries `messages`)
- `index.ts` — barrel exports

**`src/tool/builtin/search.ts`** — new `search` tool definition

**`src/scripts/backfill-embeddings.ts`** — one-time backfill for existing message embeddings

### Search Tool Interface

```typescript
// Tool parameters
type SearchToolParams = {
  query: string;                                          // required
  mode?: 'semantic' | 'keyword' | 'hybrid';              // default: 'hybrid'
  domain?: 'memory' | 'conversations' | 'all';           // default: 'all'
  limit?: number;                                         // default: 10, clamped 1-50
  start_time?: string;                                    // ISO 8601
  end_time?: string;                                      // ISO 8601
  role?: 'user' | 'assistant' | 'system' | 'tool';       // conversations only
  tier?: 'core' | 'working' | 'archival';                 // memory only
};
```

### SearchStore Port

```typescript
interface SearchDomain {
  readonly name: string;
  search(params: DomainSearchParams): Promise<ReadonlyArray<DomainSearchResult>>;
}

interface SearchStore {
  search(params: SearchParams): Promise<ReadonlyArray<SearchResult>>;
  registerDomain(domain: SearchDomain): void;
}
```

### Result Shape

```typescript
type SearchResult = {
  domain: 'memory' | 'conversations';
  id: string;
  content: string;
  score: number;
  metadata: {
    tier?: MemoryTier;
    label?: string;
    role?: string;
    conversation_id?: string;
  };
  created_at: Date;
};
```

Results are returned as a unified ranked list sorted by RRF score, each tagged with its source domain.

### Data Flow

```
Agent invokes search tool
  → Tool generates embedding (if mode is semantic or hybrid)
  → Tool calls SearchStore.search()
    → SearchStore determines target domains from params
    → Promise.all() fans out to each SearchDomain.search()
      → Each domain runs keyword CTE and/or vector CTE against its table
      → Each domain returns DomainSearchResult[] with per-result scores
    → SearchStore assigns ranks per domain, computes RRF scores
    → SearchStore merges, sorts by RRF score, returns unified list
  → Tool formats results for agent
```

### Hybrid Query Pattern

Each domain runs both searches as SQL CTEs within a single query:
- **Keyword CTE:** `plainto_tsquery('english', $query)` against the `search_vector` tsvector column, ranked by `ts_rank`
- **Vector CTE:** Cosine similarity `(1 - (embedding <=> $embedding))` against the embedding column

For `mode: 'semantic'`, only the vector CTE runs (no tsquery generation needed).
For `mode: 'keyword'`, only the keyword CTE runs (no embedding generation needed — saves a round-trip to Ollama).

RRF fusion happens in TypeScript, not SQL, because it needs to merge across domains (a memory result and a conversation result competing for the same rank position).

### Message Embedding Integration

Embedding generation hooks into `persistMessage()` in `src/agent/agent.ts`. Follows the memory manager's pattern: `embed()` with graceful null fallback on failure.

**Embedded roles:** `user` and `assistant` only. `system` messages are typically context injection; `tool` messages are structured JSON — neither embeds well.

**The tsvector column** is `GENERATED ALWAYS AS (to_tsvector('english', content)) STORED`, so all messages get keyword-indexed automatically regardless of role.

## Existing Patterns

### Followed from Codebase

- **Port/adapter boundaries:** `SearchStore` is a port interface, `createSearchStore()` returns it. Matches `MemoryStore`, `EmbeddingProvider`, `PersistenceProvider`.
- **Factory functions:** `createSearchStore()`, `createMemorySearchDomain()`, `createConversationSearchDomain()` — no classes, following `createFoo()` convention.
- **Barrel exports:** `src/search/index.ts` exports public API.
- **File pattern annotation:** Each file annotates `// pattern: Functional Core` or `// pattern: Imperative Shell`.
- **Embedding generation:** Follows `MemoryManager.generateEmbedding()` pattern — try/catch with null fallback.
- **Parallel queries:** Uses `Promise.all()` for concurrent independent queries, matching patterns in memory manager and activity context provider.
- **Vector search SQL:** Same `(1 - (embedding <=> $embedding))` cosine similarity formula as `postgres-store.ts`.

### New Patterns Introduced

- **`SearchDomain` registration:** New extensibility pattern for adding search domains. No existing equivalent — memory and conversations are the first two domains, with cross-agent messages designed as a future third.
- **RRF merge:** New scoring strategy. No existing result-merging pattern in the codebase to follow.
- **`GENERATED ALWAYS AS ... STORED` tsvector columns:** New use of PostgreSQL generated columns. No existing tsvector usage anywhere in the schema.

## Additional Considerations

**Existing `memory_read` tool:** Retained as-is. It serves a different purpose — simple semantic memory search for the common case. The new `search` tool is for broader, more configurable retrieval. Both coexist.

**Cross-agent domain extensibility:** The `SearchDomain` registration pattern makes adding a future cross-agent message domain straightforward — implement `SearchDomain`, register it, add `'cross_agent'` to the domain enum. No changes to `SearchStore` or RRF logic needed.

**HNSW index on message embeddings:** Deferred. For current message volumes, sequential vector scan is fine. If query performance degrades, add an HNSW index on `messages.embedding` in a future migration.
