# Terminal UI Design

## Summary

Constellation's agent loop currently writes to the terminal through a bare readline REPL: output is buffered until a response completes, tool calls are invisible while running, and there is no persistent status display. This design adds an opt-in terminal UI (`--tui` flag) that replaces that REPL with a rich chat interface built on Ink — the React-for-terminals library — while leaving the existing REPL entirely untouched as the default.

The central mechanism is a typed event bus that decouples the agent from its display layer. Rather than coupling output directly to stdout, the agent publishes structured events (`stream:chunk`, `tool:start`, `tool:result`, etc.) at each decision point. The Ink application subscribes to those events and renders them incrementally: text streams in character-by-character, tool calls appear inline with spinners that resolve to checkmarks, thinking content from extended-thinking models surfaces dimmed and then collapses, and a persistent status bar tracks token usage and activity state. The simple REPL can subscribe to the same bus with its own lightweight handler, meaning both modes share the same agent event contract. The only agent-side change is switching from `model.complete()` to `model.stream()` when a bus is present; all four existing model adapters already implement streaming.

## Definition of Done

An opt-in terminal UI (`--tui` flag) that replaces the bare readline REPL with a single-column enhanced chat interface built on Ink (React for CLI). The TUI shows streaming LLM responses as they arrive, displays inline tool call indicators with spinners during execution, renders thinking content from extended thinking models, and provides a persistent status bar with token usage, model name, and activity state. The existing simple REPL remains the default and is unmodified.

The integration uses an event bus architecture: the agent publishes typed events (`stream:chunk`, `tool:start`, `tool:result`, etc.) and the TUI subscribes. The simple REPL can also subscribe with its own lightweight handler. The agent loop switches from `model.complete()` to `model.stream()` when an event bus is present.

**Out of scope:** Split-pane/sidebar layouts, scrollback navigation, mouse interaction, web UI, theming/colour configuration.

## Glossary

- **Ink**: A React-based framework for building interactive CLI applications. Components render to terminal output rather than a browser DOM.
- **Event bus**: A publish/subscribe mechanism where producers emit named events and consumers register handlers for event types they care about. Decouples the agent loop from the display layer.
- **Discriminated union**: A TypeScript pattern where a shared literal field (here `type`) narrows a union type to a specific variant, enabling exhaustive type-safe event handling.
- **`AsyncIterable<StreamEvent>`**: A TypeScript interface for consuming a sequence of values asynchronously as they are produced — used here to consume the LLM token stream.
- **`model.stream()` / `model.complete()`**: Two modes on the `ModelProvider` interface. `complete()` waits for the full response; `stream()` yields incremental `StreamEvent` chunks as the model produces them.
- **`ModelProvider`**: The port interface that all LLM adapters implement. Defines the contract for sending requests and receiving responses from a model backend.
- **`AgentDependencies`**: A plain object type grouping all optional collaborators injected into the agent at construction (compactor, trace recorder, skills, and now event bus).
- **`StreamEvent`**: The low-level event type emitted by model adapters during streaming (`message_start`, `content_block_delta`, `message_stop`, etc.). The agent translates these into higher-level `AgentEvent` types before publishing to the bus.
- **Functional Core / Imperative Shell**: An architectural pattern used throughout the project. Pure functions with no side effects form the core; code that interacts with the outside world (I/O, rendering) is the shell.
- **Mutation approval**: A flow where the agent pauses and asks the user to approve, reject, or provide feedback on a proposed memory block change before writing it.
- **Compaction**: The context compression pipeline that summarises and archives older conversation turns to keep the context window within token limits.
- **Extended thinking**: An Anthropic model feature where the model externalises its reasoning as a separate `thinking` content block before producing its final response.
- **Backpressure**: Pressure that builds when a producer (the token stream) emits faster than a consumer (the renderer) can handle. The `useAgentEvents` hook batches high-frequency `stream:chunk` events to avoid excessive re-renders.
- **Barrel export**: A module pattern where `index.ts` re-exports the public API of a directory, so consumers import from the directory path rather than individual files.
- **TTY**: In Unix contexts, refers to an interactive terminal device. The `--tui` flag detects non-TTY environments (pipes, files) to fall back gracefully.

## Architecture

The TUI sits behind a typed event bus that decouples the agent's processing from display concerns. Three layers:

1. **Event bus** (`src/tui/event-bus.ts`) — typed pub/sub. `createAgentEventBus()` returns `{ publish, subscribe, unsubscribe }`. No external dependency. Events are fire-and-forget; the bus doesn't buffer or replay.

2. **Agent integration** — the agent loop in `src/agent/agent.ts` accepts an optional `AgentEventBus` in `AgentDependencies`. When present, it publishes events at each decision point and switches from `model.complete()` to `model.stream()`. When absent, behaviour is identical to today.

3. **Ink application** (`src/tui/app.tsx`) — React component tree that subscribes to the bus and renders the chat interface. Mounted by the entry point when `--tui` is passed.

### Event schema

```typescript
type AgentEvent =
  | { type: 'stream:start'; model: string; turnIndex: number }
  | { type: 'stream:chunk'; text: string; turnIndex: number }
  | { type: 'stream:thinking'; text: string; turnIndex: number }
  | { type: 'stream:end'; usage: UsageStats; stopReason: StopReason }
  | { type: 'tool:start'; toolName: string; toolId: string; input: unknown }
  | { type: 'tool:result'; toolId: string; result: string; isError: boolean }
  | { type: 'turn:start'; source: 'user' | 'event' | 'scheduled' }
  | { type: 'turn:end'; messageCount: number }
  | { type: 'compaction:start' }
  | { type: 'compaction:end'; removedTokens: number }
  | { type: 'activity:wake'; reason: string }
  | { type: 'activity:sleep' }
  | { type: 'event:received'; source: string; summary: string }
  | { type: 'error'; error: Error; context: string }
  | { type: 'mutation:request'; mutationId: string; blockId: string; proposedContent: string; reason: string | null }
  | { type: 'mutation:response'; mutationId: string; approved: boolean; feedback?: string }
```

`UsageStats` and `StopReason` are the existing types from `src/model/types.ts`. The event bus reuses them directly — no new type wrappers.

### Component tree

```
<App>
  <StatusBar />              — model name, token usage, activity state
  <ConversationView />       — scrollable message history
    <Message role="user" />
    <Message role="assistant" />
      <StreamingText />      — renders chunks incrementally
      <ThinkingIndicator />  — collapsible thinking content
    <ToolCallGroup />
      <ToolCall status="running" />   — spinner + tool name
      <ToolCall status="complete" />  — checkmark + summary
  <Divider />
  <InputArea />              — text input, disabled during processing
```

Each component subscribes to the event bus via a `useAgentEvents(bus, eventTypes[])` hook. No global state store — each component owns its display state derived from events.

### Data flow

1. User types in `<InputArea>`, hits enter.
2. Ink app calls `agent.processMessage(text)` (async, non-blocking from React's perspective).
3. Agent publishes `turn:start`, then calls `model.stream()`.
4. Stream chunks arrive → `stream:chunk` events → `<StreamingText>` renders incrementally.
5. Thinking chunks → `stream:thinking` events → `<ThinkingIndicator>` updates.
6. Stream ends → `stream:end` with usage → `<StatusBar>` updates token counts.
7. Tool use detected → `tool:start` → spinner in `<ToolCall>` → dispatch → `tool:result` → checkmark.
8. Agent may loop (tool results fed back to LLM) — steps 3-7 repeat.
9. Agent publishes `turn:end` → input re-enables.

External events (Bluesky firehose, scheduled tasks) follow the same flow but are prefixed with `event:received` and display with a different label in the conversation view.

### Mutation approval

The current mutation approval flow uses `readline.question()`. In TUI mode, the agent publishes a `mutation:request` event. The Ink app renders an inline confirmation prompt. The user's response is published back as `mutation:response`, which the agent awaits via a promise resolved by the bus.

## Existing patterns

The project follows factory functions over classes (`createFoo()` returns interface) and annotates every file with `// pattern: Functional Core` or `// pattern: Imperative Shell`. The TUI module follows both conventions:

- `createAgentEventBus()` — factory function, functional core (pure pub/sub, no side effects beyond callbacks)
- `src/tui/app.tsx` — imperative shell (renders to terminal, manages React lifecycle)
- Component files — imperative shell (React components with side effects via hooks)

The `AgentDependencies` type in `src/agent/types.ts` already uses optional dependencies (`compactor?`, `traceRecorder?`, `skills?`). Adding `eventBus?: AgentEventBus` follows this established pattern.

The `ModelProvider` interface in `src/model/types.ts` already defines `stream(request: ModelRequest): AsyncIterable<StreamEvent>`. All four adapters (Anthropic, OpenAI-compat, Ollama, OpenRouter) implement it. The TUI design consumes this existing streaming capability — no model layer changes needed.

The `StreamEvent` type (`message_start`, `content_block_start`, `content_block_delta`, `message_stop`) maps directly to `AgentEvent` stream events. The agent translates between them during publishing.

## Additional considerations

**Dependency additions:** `ink`, `react`, `ink-text-input`, `ink-spinner`, `chalk`. These are dev/runtime dependencies for the TUI module only. The simple REPL path doesn't load them.

**Streaming assembly:** When the agent switches to `model.stream()`, it needs to assemble a `ModelResponse` from the stream events (accumulate text chunks, collect tool use blocks from deltas, capture usage stats from `message_stop`). This assembly logic lives in the agent, not the TUI — the TUI only cares about display events, not about reconstructing the response.

**Backpressure:** Ink re-renders on state changes. High-frequency `stream:chunk` events could cause excessive renders. The `useAgentEvents` hook should batch chunks within animation frames (requestAnimationFrame equivalent, or Ink's built-in batching) to keep rendering smooth.
