# Skills System — Human Test Plan

## Prerequisites

- PostgreSQL running: `docker compose up -d`
- Migrations applied: `bun run migrate`
- All automated tests passing: `bun test` (103 pass, 0 fail)
- A valid `config.toml` in the project root with embedding provider configured (OpenAI or Ollama at `192.168.1.6:11434`)
- At least one builtin skill present in `./skills/*/SKILL.md`

## Phase 1: Composition Root — Skills Enabled

| Step | Action | Expected |
|------|--------|----------|
| 1.1 | Add a `[skills]` section to `config.toml` with `builtin_dir = "./skills"` and `user_dir = "./user-skills"` | File saved with valid TOML |
| 1.2 | Run `bun run start` | Console output contains a line like `skills loaded (N skills)` where N > 0 |
| 1.3 | Send a message to the REPL that relates to a known builtin skill topic | Agent response references or uses the skill's instructions; system prompt contains `## Active Skills` |
| 1.4 | Send an unrelated message (e.g., "what is 2+2?") | No `## Active Skills` section should appear (no skills match a trivial maths question) |

## Phase 2: Composition Root — Skills Disabled

| Step | Action | Expected |
|------|--------|----------|
| 2.1 | Comment out or remove the entire `[skills]` section from `config.toml` | File saved without `[skills]` |
| 2.2 | Run `bun run start` | No skill-related console output. No errors. Agent starts normally |
| 2.3 | Send a message to the REPL | Agent responds normally. No `## Active Skills` section in the system prompt |

## Phase 3: Skill-Defined Tools Registration

| Step | Action | Expected |
|------|--------|----------|
| 3.1 | Create a test skill at `./skills/persona-gen/SKILL.md` with a `tools` entry in frontmatter | File saved |
| 3.2 | Restart the daemon with `[skills]` enabled in `config.toml` | Console output shows the skill loaded |
| 3.3 | Ask the agent: "Use the generate-persona tool with archetype 'trickster'" | Agent invokes the tool. Tool result contains `[Skill: persona-gen]` followed by the skill's markdown body |
| 3.4 | Verify the agent incorporates the skill body instructions in its subsequent response | Agent acts on the skill's instructions |

## Phase 4: Skill Management via Agent Tools

| Step | Action | Expected |
|------|--------|----------|
| 4.1 | Ask the agent: "List all skills" | Agent calls `skill_list`; response shows all loaded skills with names, descriptions, sources, and tags |
| 4.2 | Ask the agent: "List only user skills" | Agent calls `skill_list` with `source: 'user'`; only user-sourced skills appear |
| 4.3 | Ask the agent: "Read the skill called [known-skill-name]" | Agent calls `skill_read`; response includes full body content and any companion files |
| 4.4 | Ask the agent: "Read the skill called nonexistent-skill-xyz" | Agent calls `skill_read`; response indicates skill not found |
| 4.5 | Ask the agent: "Create a new skill called test-manual with description 'Manual test skill' and body 'This is a manually created skill.'" | Agent calls `skill_create`; response confirms creation. Verify `./user-skills/test-manual/SKILL.md` exists on disk |
| 4.6 | Ask the agent: "Create a skill called Invalid Name" | Agent calls `skill_create`; response reports invalid name format error |
| 4.7 | Ask the agent: "Update the skill test-manual with description 'Updated manual skill' and body 'Updated body content.'" | Agent calls `skill_update`; response confirms update. Verify file on disk reflects changes |
| 4.8 | Ask the agent: "Update the skill [builtin-skill-name]" (use a known builtin skill name) | Agent calls `skill_update`; response reports error that builtin skills cannot be updated |

## End-to-End: Skill Lifecycle

1. Start the daemon with `[skills]` enabled and `similarity_threshold = 0.3`
2. Create a user skill via the agent: "Create a skill called code-review with description 'Guidelines for reviewing TypeScript code' and body '## Code Review Guidelines\n\n1. Check for type safety\n2. Verify error handling\n3. Review naming conventions'"
3. Verify the skill file exists at `./user-skills/code-review/SKILL.md`
4. Send a message: "Please review this TypeScript function: `function add(a, b) { return a + b; }`"
5. Verify the `## Active Skills` section in the system prompt includes `### code-review`
6. Verify the agent's response reflects the code review guidelines from the skill body
7. Update the skill: "Update the skill code-review with body '## Updated Guidelines\n\n1. Check for strict mode\n2. Verify tests exist'"
8. Send another code review message and verify the updated guidelines appear
9. Restart the daemon and verify the skill persists and is loaded on startup

## End-to-End: Change Detection

1. Start the daemon, note the list of loaded skills
2. Stop the daemon
3. Modify the body of one builtin skill's `SKILL.md` file on disk
4. Restart the daemon
5. Observe console output — only the modified skill should be re-embedded
6. Skills with unchanged content should not trigger embedding calls
