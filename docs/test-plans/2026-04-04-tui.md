# Human Test Plan: Terminal UI (TUI)

## Prerequisites

- PostgreSQL 17 with pgvector running (`docker compose up -d`)
- All dependencies installed (`bun install`)
- `bun test` passing (all automated tests green)
- Valid `config.toml` with at least one LLM provider configured
- Terminal emulator that supports 256 colours and Unicode (iTerm2, Ghostty, kitty, etc.)

## Phase 1: TUI Flag Detection (tui.AC8)

| Step | Action | Expected |
|------|--------|----------|
| 1 | Run `bun run start --tui` in an interactive terminal | Ink-based TUI launches. Status bar at top with model name and `0` token counts, conversation area in the middle (empty), and `>` input prompt at the bottom. No readline `> ` prompt. |
| 2 | Type a short message (e.g., "hello") and press Enter | Message appears with `You:` label. Assistant response streams incrementally. Input shows "Processing..." during generation. |
| 3 | Press Ctrl+C | TUI exits cleanly without stacktrace |
| 4 | Run `bun run start` (no `--tui` flag) | Standard readline REPL launches with `> ` prompt. No Ink components visible. |
| 5 | Type "hello" and press Enter in REPL mode | Agent responds inline, same as before TUI work |
| 6 | Run `echo "hello" \| bun run start --tui` | Warning on stderr containing "not running in a TTY environment". REPL mode activates or exits gracefully. |

## Phase 2: Streaming Display (tui.AC3)

| Step | Action | Expected |
|------|--------|----------|
| 1 | In TUI mode, send a message requiring a long response (e.g., "write a haiku about each season") | Text streams character-by-character or chunk-by-chunk. |
| 2 | Observe the status bar during and after streaming | After response: token counts update from `0` to actual values (e.g., `1234↓ 567↑`). |
| 3 | Observe the input area during streaming | Shows "Processing..." while generating. After completion, `>` prompt reappears. |
| 4 | Send a second message after the first completes | Both user+assistant pairs visible as distinct entries with `You:` and `Assistant:` labels. Token counts increase cumulatively. |

## Phase 3: Tool Call Display (tui.AC4)

| Step | Action | Expected |
|------|--------|----------|
| 1 | Send a message that triggers tool use (e.g., "remember that my favourite colour is blue") | Spinner with tool name visible while running. Checkmark with tool name and result summary after completion. |
| 2 | Observe tool display after turn completes | Tool calls collapse into a summary line (e.g., "3 tool calls"). |
| 3 | Trigger a tool that might fail (e.g., web fetch to nonexistent URL) | Error indicator (cross) with error message instead of checkmark. |

## Phase 4: Thinking Display (tui.AC5)

*Requires a model with extended thinking support.*

| Step | Action | Expected |
|------|--------|----------|
| 1 | Send a complex message that triggers extended thinking | Dimmed thinking text appears with "Thinking:" label, separate from main response. |
| 2 | Observe thinking display after turn completes | Collapses into summary showing character count (e.g., "Thinking (243 chars)"). |

## Phase 5: Mutation Approval (tui.AC6)

*Requires triggering a core memory mutation.*

| Step | Action | Expected |
|------|--------|----------|
| 1 | Send a message that triggers a memory block update | Inline mutation prompt shows block ID, proposed content, and approval options (`y`/`n`/`f`). |
| 2 | Press `y` to approve | Mutation applied. Agent continues. Prompt disappears. |
| 3 | Trigger another mutation and press `n` | Mutation rejected. Agent acknowledges rejection. |
| 4 | Trigger another mutation, press `f`, type feedback, press Enter | Mutation rejected with feedback. Agent incorporates feedback. |

## Phase 6: System Events (tui.AC7)

*Some require specific configurations.*

| Step | Action | Expected |
|------|--------|----------|
| 1 | If DataSource configured, wait for external event | System event with `[source]` label and summary text. |
| 2 | Trigger compaction (lower `context_budget` or send many messages) | "Compacting context" indicator, then token savings message. |
| 3 | If activity cycle configured, wait for sleep/wake transition | Sleep/wake messages appear. Status bar indicator changes. |

## End-to-End: Full Conversation Lifecycle

1. Launch TUI mode: `bun run start --tui`
2. Verify initial state: status bar with model name and `0↓ 0↑`, empty conversation, `>` prompt
3. Send "hello, what can you do?" — observe streaming, token update, input disable/re-enable
4. Send "remember that I prefer dark mode" — observe tool calls, mutation prompt if applicable, streaming
5. Send "what do you remember about my preferences?" — observe tool use, streaming with stored preference
6. Verify 3 user messages and 3 assistant responses as distinct entries
7. Verify cumulative token counts across all turns
8. Press Ctrl+C — clean exit
